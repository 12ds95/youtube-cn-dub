# 翻译质量优化：GT 对比 + 跨段错位修复

## 背景

Pipeline 翻译质量不理想，通过下载 YouTube 社区中文字幕作为 Ground Truth 进行对比评估，发现系统性问题并逐步优化。

## 调研参考

- **Amazon Isometric MT**: ±10% 字符数为合规阈值，"length control comes at a cost to quality"
- **LLM prompting for isometric**: 92% 合规率，BLEU 仅下降 1-2 分，关键是在 prompt 中嵌入显式长度指令
- **SSPO (ACL 2025)**: 关键洞察——这是一个逐段偏好优化问题，不是全局单目标优化

## 实施的修改

### 1. Pass 2 prompt: "润色专家" → "审校专家"
- 去除 "口语化" 要求，改为 "准确、通顺、自然的标准中文"
- 加入 "如果已通顺，直接保持原样" 防止过度改写

### 2. 显式长度指令 (基于调研)
- Pass 1 char hints: 精确范围 `≈center字(lo-hi)` (75%-130%)
- 指令从 "仅供参考可超出" 改为 "请尽量控制在范围内"
- Single fallback: 恢复 `目标≈N字` 格式

### 3. 去除双重 isometric
- Pass 1 `isometric=0`，仅在 Pass 2 后运行一次

### 4. 扩展上下文窗口
- 回看 6→10 段，前瞻 3→5 段

### 5. 句子碎片标注
- 检测 Whisper 切碎的短碎片（<6 词 AND <2s）
- 在 prompt 中标注 `{续}` 标记，提示 LLM 均匀分配句意

## 实验结果

### 基线 (修改前)
```
TOTAL: 4440 segs, ellipsis=165, telegram=175, dup=72
info_ratio=0.52, win_info=0.71, jieba=1.24, 适配%=38%
```

### 优化后 (3 个视频重新翻译)
```
TOTAL: 4483 segs, ellipsis=158, telegram=213, dup=101
info_ratio=0.54, win_info=0.75, jieba=1.31, 适配%=36%
```

### 逐视频对比 (仅重新翻译的视频)

| 视频 | info_ratio | win_info | jieba | 说明 |
|---|---|---|---|---|
| zjMuIxRvygQ | 0.60→0.63 | 0.79→0.82 | 1.18→1.22 | 无碎片组，纯 prompt 改进 |
| Epidemic | 0.27→0.37 | 0.47→0.64 | 1.79→2.19 | 218 组碎片标注，信息比 +37% |
| CLT | 0.40→0.48 | 0.59→0.71 | 1.51→1.76 | 97 组碎片标注，信息比 +20% |

### 关键发现

1. **跨段错位是主要问题，不是翻译质量**: 3段窗口 info_ratio 远高于逐段 info_ratio (0.75 vs 0.54)，说明 97% 的 "信息丢失" 其实是跨段分布不均
2. **两个优化目标在不同段上失败**: 质量差的段和时长不匹配的段很少重叠（Pareto 分析）
3. **极端碎片化视频（Epidemic/CLT）的 GT 结构完全不同**: GT 是完整句字幕，我们是 764 个 3-5 词碎片，逐段比较天然不公平
4. **合并+拆分方案失败**: 尝试将碎片合并翻译再按时长比例拆回，导致 dup 爆增和 jieba 失控
5. **标注方案更安全**: 只在 prompt 中标注碎片关系，让 LLM 自行分配，不改变段结构

## 未来方向

- 碎片合并应在 Whisper 转录后 / 翻译前进行（merge segments），而非翻译中处理
- 评估体系的 dup 阈值（6字符）过于敏感，应提高到 8-10 字符或使用语义去重
- 对极端碎片化视频，考虑启用 `nlp_segmentation` (spaCy) 进行预合并
