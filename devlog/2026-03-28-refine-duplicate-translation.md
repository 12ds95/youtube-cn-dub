# 迭代优化(refine)导致翻译语句重复 Bug 排查与修复

日期：2026-03-28
影响范围：`pipeline.py` — `_refine_with_llm()`, `_align_tts_to_timeline()`, 新增 `_expand_with_llm()`, `deduplicate_segments()`, `_is_duplicate_of_neighbors()`
测试视频：3Blue1Brown 四元数讲解，输出目录 `output/32884a7ba3d/`


## 用户反馈的现象

最终视频中部分中文配音和字幕出现语句重复——相邻两段说的是几乎相同的话。


## 排查过程

### 第一步：检查 segments_cache.json 是否有重复

对最终缓存做全量扫描（连续相同、非连续相同、子串包含）：

```
总片段数: 72
连续重复英文: 0
连续重复中文: 0
非连续重复英文: 0
非连续重复中文: 0
```

精确匹配层面无重复。但逐段审查字幕发现明显的近义重复：

```
#4 ZH: 这次合作对我们双方而言，都是一次新尝试。
#5 ZH: 这次合作对我们都是新尝试。

#7 ZH: 这无疑是我参与过最酷的项目之一。
#8 ZH: 这无疑是我有幸参与过的最酷炫的项目之一。

#13 ZH: 其中一个重要原因，尤其是对程序员而言，
#14 ZH: 尤其对程序员而言，它能优雅高效地描述三维朝向。
```

注意：segments_cache 中编号从 0 开始，SRT 字幕从 1 开始，对应关系为 cache #3=#SRT4, cache #6=#SRT7 等。

### 第二步：确认英文转录无重复

检查对应的英文文本——每段英文内容完全不同，Whisper 转录没有问题。

### 第三步：对比初始翻译(iter_0) vs 最终翻译

```
--- #3 & #4 ---
  iter0 ZH #3: 不过，这次合作对我们双方而言，都是一次略有不同的尝试。
  iter0 ZH #4: 而让这些"可探索视频"成为可能的所有网页开发工作，
  final ZH #3: 这次合作对我们双方而言，都是一次新尝试。
  final ZH #4: 这次合作对我们都是新尝试。

--- #6 & #7 ---
  iter0 ZH #6: 因为它真正令人惊叹之处，唯有亲身体验才能体会。
  iter0 ZH #7: 这无疑是我有幸参与过的最酷炫的项目之一。
  final ZH #6: 这无疑是我参与过最酷的项目之一。
  final ZH #7: 这无疑是我有幸参与过的最酷炫的项目之一。
```

**关键发现：初始翻译(iter_0)各段内容不同，重复是迭代优化过程引入的。**

### 第四步：追踪迭代变更记录定位具体轮次

```
iter_1: #6 "唯有亲身体验才能体会" → "这无疑是我参与过最酷的项目之一" （抄了 #7）
iter_1: #14 "就在于它能为描述三维空间中的朝向..." → "尤其对程序员而言，它能优雅高效地描述三维朝向" （抄了 #13 开头）
iter_2: #4 "所有网页开发工作完全归功于本" → "这次合作对我们双方都是新尝试" （抄了 #3）
iter_4: #4 "这次合作对我们双方都是新尝试" → "这次合作对我们都是新尝试" （继续精简，但仍是 #3 内容）
```

### 根因定位

**迭代优化的精简 prompt 给了上下文（上文/下文），LLM 在缩短翻译时偷懒复制了相邻段的内容，而不是忠实精简当前英文原文的翻译。** 采纳逻辑只检查了 `len(新) < len(旧)`，没有检查是否与相邻段重复。

### 修改 1：强化精简 prompt

在 system_prompt 中增加：
- "必须忠实翻译当前英文原文，不得偏离原文含义"
- "严禁与上文或下文重复——精简后的译文不能和相邻段落说同样的话"

## 经验总结

1. **LLM 精简时给上下文是双刃剑**：上下文帮助保持语义连贯，但也给了 LLM 偷懒的机会。必须在 prompt 中明确禁止抄相邻段，并在采纳时做程序化验证。

2. **字符重叠率比最长公共子串更适合中文**：中文近义改写往往打散了字符顺序但使用相同字符，字符级 Counter 交集能更好捕捉这种相似性。

3. **排查重复问题要逐层追溯**：直觉可能指向 Whisper 转录或 LLM 翻译，但实际根因在迭代优化环节。逐层对比 iter_0 → iter_N → final 快速锁定问题引入的具体轮次和变更记录。
